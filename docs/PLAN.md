# Проект обучения LLM — План

> Статус: черновик. Подлежит уточнению с заказчиком.

---

## 1. Цели проекта

- **Что строим:** собственная LLM (Large Language Model) — либо с нуля (pre-train), либо fine-tune открытой модели под конкретный домен/задачи.
- **Ключевые вопросы, требующие уточнения:**
  - Размер модели? (1B / 7B / 13B / 70B+)
  - С нуля или fine-tune существующей (Llama, Mistral, Qwen, DeepSeek)?
  - Целевой домен: код, медицина, русский язык, общий ассистент?
  - Бюджет: оборудование (GPU-кластер / облако), сроки.

---

## 2. Этапы проекта

### Этап 0: Исследование и дизайн (1–2 недели)

- [ ] Обзор SOTA-моделей, доступных датасетов и архитектур.
- [ ] Выбор базовой стратегии: pre-train vs fine-tune.
- [ ] Определение метрик качества (perplexity, MMLU, HumanEval, Chatbot Arena-style).
- [ ] Оценка вычислительных потребностей (GPU-часы, память).
- [ ] Выбор стека: PyTorch + HF Transformers / Megatron-LM / DeepSpeed / FSDP.

### Этап 1: Подготовка данных (2–4 недели)

- [ ] **Сбор корпуса** — CommonCrawl, Wikipedia, книги, код (The Stack), специализированные источники.
- [ ] **Очистка и фильтрация:**
  - Дедупликация (MinHash, SimHash)
  - Фильтрация по качеству (perplexity-классификатор, длина, язык)
  - Удаление PII и токсичного контента
  - Фильтрация шаблонного/boilerplate текста
- [ ] **Токенизация:**
  - Выбор/обучение токенизатора (BPE / SentencePiece / Tiktoken)
  - Расчёт размера словаря (32k–128k токенов)
  - Предварительная токенизация корпуса для быстрой загрузки
- [ ] **Перемешивание и упаковка в последовательности** (packing для эффективного обучения).

### Этап 2: Pre-training (2–8 недель, зависит от размера)

- [ ] **Конфигурация модели:**
  - Архитектура: decoder-only Transformer (GPT-style)
  - Кол-во слоёв, голов внимания, размер эмбеддингов
  - Позиционные эмбеддинги: RoPE / ALiBi
  - Нормализация: Pre-LN / RMSNorm
  - Активация: GELU / SwiGLU
- [ ] **Настройка обучения:**
  - Оптимизатор: AdamW с warmup и cosine decay
  - Mixed precision: bf16/fp16
  - Градиентный clipping
  - Batch size и gradient accumulation
- [ ] **Распределённое обучение:**
  - DeepSpeed ZeRO Stage 1/2/3 или FSDP
  - Pipeline parallelism (если модель > одного узла)
  - Контрольные точки (checkpointing) каждые N шагов
- [ ] **Мониторинг:** loss, градиентные нормы, throughput (токенов/сек).

### Этап 3: Fine-tuning / SFT (1–2 недели)

- [ ] **Supervised Fine-Tuning (SFT)** — инструктивные датасеты:
  - OpenAssistant, Dolly, LIMA, UltraChat, WildChat
  - Собственные размеченные данные (если есть)
- [ ] Формат: чат-шаблон (ChatML, Llama-chat, etc.)
- [ ] LoRA / QLoRA для экономии памяти (если оборудование ограничено).
- [ ] Полноценный fine-tuning (full-weight) если ресурсов достаточно.

### Этап 4: Alignment / RLHF / DPO (1–3 недели)

- [ ] **Сбор предпочтений:**
  - Генерация ответов моделью SFT
  - Ранжирование людьми или LLM-as-judge
- [ ] **Reward Model (RM):** обучение модели вознаграждения на парах предпочтений.
- [ ] **PPO / RLHF:** proximal policy optimization с KL-штрафом к SFT.
- [ ] **DPO (Direct Preference Optimization):** более простая альтернатива RLHF без reward-модели.
- [ ] **Оценка безопасности:** red-teaming, отказ от вредных запросов.

### Этап 5: Оценка и бенчмарки (1 неделя)

- [ ] **Стандартные бенчмарки:**
  - MMLU, HellaSwag, ARC, TruthfulQA, GSM8K
  - HumanEval / MBPP (для кода)
  - MT-Bench, Chatbot Arena Elo
- [ ] **Доменные метрики** (под задачу).
- [ ] **Оценка задержки и пропускной способности** инференса.

### Этап 6: Оптимизация инференса (1–2 недели)

- [ ] **Квантизация:** GPTQ, AWQ, GGUF (llama.cpp), bitsandbytes.
- [ ] **Компиляция:** TensorRT-LLM, vLLM, llama.cpp.
- [ ] **KV-cache оптимизация:** paged attention, continuous batching.
- [ ] **Бенчмарк latency/throughput.**

### Этап 7: Деплой (1 неделя)

- [ ] Упаковка модели (HF Hub / приватный репозиторий).
- [ ] REST API сервер (vLLM / TGI / FastAPI).
- [ ] Мониторинг в production.
- [ ] CI/CD для обновления модели.

## 4. Оценка ресурсов (грубо)

| Размер модели | GPU-часы pre-train | GPU для инференса (bf16) |
|--------------|--------------------|------------------------|
| ~1B | ~3K (A100-80GB) | 1× A10 |
| ~7B | ~50K (A100-80GB) | 1× A100 |
| ~13B | ~100K | 2× A100 |
| ~70B | ~1M+ | 4-8× A100 |

---

## 5. Риски

1. **Данные низкого качества** → плохая модель. Митигация: строгая фильтрация.
2. **Нестабильность обучения** (loss spikes). Митигация: gradient clipping, перезапуск с чекпоинта.
3. **Перерасход бюджета.** Митигация: early-stopping на основе validation loss.
4. **Безопасность модели.** Митигация: red-teaming, фильтры, alignment.
5. **Юридические риски** (авторские права на данные). Митигация: аудит источников.
